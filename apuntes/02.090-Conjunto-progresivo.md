# Diseñando una clase `Conjunto` — de array a LINQ

Este tutorial recorre la construcción de una clase `Conjunto<T>` en seis etapas. Cada etapa resuelve una limitación concreta de la anterior, y cada refactor introduce una herramienta del lenguaje que vale la pena conocer por sí misma: **arrays**, **composición**, **genéricos**, **interfaces**, **`IEnumerable<T>`** y finalmente **LINQ**.

La idea es doble: terminamos con una implementación corta y expresiva, pero además en el camino se ve *por qué* existen esas herramientas. No son decoración — cada una elimina un problema real que teníamos en la etapa anterior.

Un conjunto matemático tiene tres propiedades: sus elementos son únicos (no hay duplicados), no tienen orden, y podemos preguntar rápido si algo pertenece o no. Vamos a implementar eso progresivamente.

---

## Etapa 1 — Conjunto con array interno

Empezamos por lo más básico: un conjunto de enteros respaldado por un `int[]`. No usamos ninguna estructura de datos de la biblioteca estándar — queremos ver la mecánica.

```csharp
class Conjunto {
    int[] elementos;
    int count;

    public Conjunto(int capacidad = 10) {
        elementos = new int[capacidad];
        count = 0;
    }

    public void Agregar(int valor) {
        if (Contiene(valor)) { return; }   // evita duplicados
        elementos[count++] = valor;
    }

    public void Eliminar(int valor) {
        for (int i = 0; i < count; i++) {
            if (elementos[i] == valor) {
                elementos[i] = elementos[count - 1]; // swap con el último
                count--;
                return;
            }
        }
    }

    public bool Contiene(int valor) {
        for (int i = 0; i < count; i++) {
            if (elementos[i] == valor) { return true; }
        }
        return false;
    }

    // Devuelve una copia de los elementos actuales del conjunto
    public int[] Elementos {
        get { 
            int[] copia = new int[count];
            Array.Copy(elementos, copia, count);
            return copia; 
        }
    }
    public int Count => count;

    public override string ToString() => $"{{{string.Join(", ", elementos)}}}";
}
```

Tres detalles que vale la pena mirar:

**Eliminar con swap.** En lugar de correr todos los elementos para "tapar el hueco", reemplazamos el elemento a eliminar por el último del array y decrementamos `count`. Es O(1) en vez de O(n), y como el conjunto no tiene orden, podemos permitirnos reordenar.

**`count` vs largo del array.** El array tiene capacidad fija, pero el conjunto puede tener menos elementos. Por eso todo recorrido va hasta `count`, no hasta `elementos.Length`.

**Elementos devuelve una copia.** No queremos que el consumidor modifique el array interno, así que devolvemos una copia con solo los elementos válidos.

### Operaciones entre conjuntos

A partir de los métodos básicos, podemos escribir operaciones como unión e intersección:
Para poder realizar esta operación, necesitamos acceder a los elementos de ambos conjuntos. Para eso, usamos la propiedad `Elementos` que devuelve una copia del array con los elementos actuales del conjunto. Luego, iteramos sobre esos elementos para construir el resultado.

```csharp
partial class Conjunto {
 
    public static Conjunto Union(Conjunto a, Conjunto b) {
        var r = new Conjunto();
        foreach (var e in a.Elementos) { r.Agregar(e); }
        foreach (var e in b.Elementos) { r.Agregar(e); }
        return r;
    }

    public static Conjunto Interseccion(Conjunto a, Conjunto b) {
        var r = new Conjunto();
        foreach (var e in a.Elementos) {
            if (b.Contiene(e)) { r.Agregar(e); }
        }
        return r;
    }

    // Sobrecarga de operadores para facilitar la sintaxis de unión e intersección
	public static Conjunto operator |(Conjunto a, Conjunto b) =>Union(a, b);
	public static Conjunto operator &(Conjunto a, Conjunto b) =>Interseccion(a, b);
}
```

La **sobrecarga de operadores** `|` y `&` imita la notación de conjuntos de lógica (`∪` y `∩`), y permite escribir `a | b | c` en lugar de anidar llamadas.

### El problema

Esta clase funciona, pero tiene ruido de implementación por todos lados: manejamos capacidad, recorridos manuales. Y si el array se llena estamos en problema. Todo ese código no habla del *concepto* de conjunto — habla de arrays.

## Etapa 2 — Composición con `List<int>`

En vez de administrar un array nosotros, delegamos en `List<int>`, que ya resuelve el crecimiento dinámico, la búsqueda y la eliminación:

```csharp
class Conjunto {
    List<int> elementos = new();

    public void Agregar(int valor) {
        if (Contiene(valor)) { return; }
        elementos.Add(valor);
    }

    public void Eliminar(int valor) {
        if(!Contiene(valor)) { return; }
        elementos.Remove(valor);
    }

    public bool Contiene(int valor) {
        foreach (var e in elementos) {
            if (e == valor) { 
                return true; 
            }
        }
        return false;
    }

    public List<int> Elementos => elementos.ToList(); // devuelve una copia
    public int Count => elementos.Count;
    
    public override string ToString() => $"{{{string.Join(", ", elementos)}}}";
}
```

Esto es **composición**: `Conjunto` no *es* una lista, pero *usa* una internamente. La lista resuelve el "cómo guardo los datos" y nosotros nos ocupamos de lo que agrega valor: la regla de unicidad.

> **Composición vs herencia.** Podríamos haber hecho `class Conjunto : List<int>`, pero sería un error de modelado. Una lista permite duplicados y acceso por índice; un conjunto no. Al heredar expondríamos `Add`, `Insert`, `this[int]` con semánticas que rompen la invariante. Con composición, el `Conjunto` decide qué operaciones de la lista expone y bajo qué reglas.

### El problema

Resolvimos un problema, ganamos otro: esta clase solo sirve para `int`. Si mañana queremos un conjunto de `string` o de `Alumno`, tenemos que duplicar todo el código cambiando el tipo. Eso es exactamente el problema que resuelven los genéricos.

## Etapa 3 — Genéricos: `Conjunto<T>`

Un tipo genérico es una clase con un "hueco" para el tipo, que se completa al instanciarla. En vez de escribir `ConjuntoInt`, `ConjuntoString`, `ConjuntoAlumno`, escribimos una sola vez `Conjunto<T>`:

```csharp

// Si teniamos que tener conjunto de distintos tipos debíamos escribir algo así:
class ConjuntoInt {
    List<int> elementos = new List<int>();

    public void Agregar(int valor)  { ... }
    public void Eliminar(int valor) { ... }
    public bool Contiene(int valor) { ... }
    public List<int> Elementos =>   { ... }
    public int Count => elementos.Count;
}

class ConjuntoString {
    List<string> elementos = new List<string>();

    public void Agregar(string valor)  { ... }
    public void Eliminar(string valor) { ... }
    public bool Contiene(string valor) { ... }
    public IList<string> Elementos =>  { ... }
    public int Count => elementos.Count;
}

// Con genéricos, una sola clase sirve para todos los tipos: 

public class Conjunto<T> where T : IEquatable<T> {
    List<T> elementos = new List<T>();

    public void Agregar(T valor) {
        if (Contiene(valor)) { return; }
        elementos.Add(valor); 
    }

    public void Eliminar(T valor) {
        if(!Contiene(valor)) { return; }
        elementos.Remove(valor); 
    }

    public bool Contiene(T valor) {
        foreach (var e in elementos) {
            if (e.Equals(valor)) { 
                return true; 
            }
        }
        return false;
    }

    public List<T> Elementos => elementos.ToList();
    public int Count => elementos.Count;

    public override string ToString() => $"{{{string.Join(", ", elementos)}}}";
}

```

Ahora podemos hacer:

```csharp

class Alumno(string Nombre, int Legajo) : IEquatable<Alumno> {
    public bool Equals(Alumno? otro) {
        if (otro is null) { return false; }
        return this.Legajo == otro.Legajo;   // igualdad = mismo legajo
    }
    
    // Para Dictionary, HashSet, etc. — consistente con Equals
    public override int GetHashCode() => Legajo.GetHashCode(); // Para Dictionary, HashSet, etc. — 
}

var alumnos  = new Conjunto<Alumno>(); // Lo usamos en un tipo que no existia al escribir la clase
var enteros  = new Conjunto<int>();
var palabras = new Conjunto<string>();
```

### La restricción `where T : IEquatable<T>`

`where T : IEquatable<T>` se llama **restricción genérica** (*generic constraint*). Le decimos al compilador que `T` no puede ser cualquier cosa: tiene que ser un tipo que sepa compararse consigo mismo.

¿Por qué? Porque `Contiene` necesita preguntar "¿este elemento es igual a aquel?". Sin la restricción, `T` podría ser cualquier objeto y la comparación caería en `Object.Equals`, que por defecto compara por referencia — dos `Alumno` con los mismos datos se verían como distintos.

El conjunto va a manejar la coleccion mediante la lista, pero no tiene forma de saber y un elemento es igual a otro. Al exigir `IEquatable<T>`, garantizamos que el tipo `T` implementa un método `Equals(T otro)` que define la igualdad de forma adecuada para ese tipo.

### Interface `IConjunto<T>` y polimorfismo

También definimos una interface:

```csharp
interface IConjunto<T> where T : IEquatable<T> {
    void Agregar(T valor);
    void Eliminar(T valor);
    bool Contiene(T valor);
    IList<T> Elementos { get; }
    int Count { get; }
}
```

Con esto, una función como `Union` puede recibir *cualquier* implementación de `IConjunto<T>`:

```csharp
static IConjunto<T> Union<T>(IConjunto<T> a, IConjunto<T> b) where T : IEquatable<T> {
    var r = new Conjunto<T>();
    foreach (var e in a.Elementos) { r.Agregar(e); }
    foreach (var e in b.Elementos) { r.Agregar(e); }
    return r;
}
```

Si más adelante implementamos `ConjuntoOrdenado<T>` o `ConjuntoHash<T>`, `Union` los mezcla sin cambios.

### El problema

Si hago esto:

```csharp
record Alumno(string Nombre, int Legajo);
var c = new Conjunto<Alumno>();
c.Agregar(new Alumno("Ana", 20));
c.Agregar(new Alumno("Ani", 20)); // ¿se agrega o no?
```

Queremos que *no* se agregue, porque tienen el mismo legajo. Pero `Equals` por defecto compara todos los campos del record, así que para C# son distintos. Necesitamos decirle *qué significa* que dos alumnos sean iguales.

---

## Etapa 4 — `IEquatable<T>`: definir la igualdad

Implementamos `IEquatable<Alumno>` en el record y sobrescribimos `Equals` y `GetHashCode`:

```csharp
class Alumno(string Nombre, int Legajo) : IEquatable<Alumno> {
    public bool Equals(Alumno? otro) {
        if (otro is null) { return false; }
        return this.Legajo == otro.Legajo;   // igualdad = mismo legajo
    }

    public override int GetHashCode() => Legajo.GetHashCode(); // Para Dictionary, HashSet, etc. —
}
```

Ahora dos alumnos con el mismo legajo se consideran el mismo elemento, sin importar el nombre.

> **Regla de oro:** siempre que sobrescribas `Equals`, sobrescribí también `GetHashCode` de forma **consistente**. Dos objetos iguales deben devolver el mismo hash. Si no, estructuras como `HashSet<T>` o `Dictionary<TKey, TValue>` se rompen silenciosamente: guardás algo y no lo encontrás nunca más.

### Simplificación en `Contiene`

Como `IList<T>.Contains` ya usa el `Equals` de cada elemento internamente, podemos reemplazar el foreach manual por una línea:

```csharp
public bool Contiene(T valor) => elementos.Contains(valor);
```

### El problema

Todavía nos queda una fricción en el uso. Cuando queremos recorrer el conjunto, tenemos que ir a través de la propiedad `Elementos`:

```csharp
foreach (var a in clase.Elementos) { ... }  // funciona, pero feo
foreach (var a in clase)           { ... }  // esto no compila
```

Además, `Elementos` devuelve `List<T>` — expone un detalle de implementación innecesario. Si cambiamos la estructura interna (a un `HashSet<T>`, por ejemplo), la interfaz pública se rompe.

## Etapa 5 — `IEnumerable<T>`: hacer el conjunto iterable

Queremos que `foreach` funcione directamente sobre el conjunto. La forma idiomática de lograrlo es devolver `IEnumerable<T>` en lugar de `List<T>`:

El tipo `IEnumerable<T>` representa una secuencia de elementos que se pueden recorrer, pero no necesariamente modificar. Al exponer `IEnumerable<T>`, el consumidor puede iterar sobre los elementos del conjunto sin tener acceso directo a la estructura interna, lo que protege la invariante de unicidad.

```csharp
interface IConjunto<T> where T : IEquatable<T> {
    void Agregar(T valor);
    void Eliminar(T valor);
    bool Contiene(T valor);
    IEnumerable<T> Elementos { get; }
}

class Conjunto<T> : IConjunto<T> where T : IEquatable<T> {
    List<T> elementos = new List<T>();

    public void Agregar(T valor) {
        if (!Contiene(valor)) { elementos.Add(valor); }
    }

    public void Eliminar(T valor) => elementos.Remove(valor);
    public bool Contiene(T valor) => elementos.Contains(valor);

    public IEnumerable<T> Elementos => elementos;
    public int Count => elementos.Count;

    public override string ToString() => $"{{{string.Join(", ", elementos)}}}";
}
```

### ¿Por qué `IEnumerable<T>` y no `List<T>`?

Tres razones:

1. **Encapsulación.** Al exponer `IEnumerable<T>` el consumidor solo puede recorrer, no puede hacer `.Add()` ni `[i] = x` sobre la lista interna. La invariante de unicidad queda protegida.
2. **Libertad de implementación.** Si mañana cambiamos `List<T>` por `HashSet<T>` o un array, la firma pública no cambia.
3. **Composición con LINQ.** Todo LINQ opera sobre `IEnumerable<T>`. Devolver eso abre la puerta a la etapa 6.

### `foreach` directo sobre el conjunto

Para que `foreach (var a in clase)` funcione, el propio `Conjunto<T>` tiene que ser enumerable, no solo exponer una propiedad enumerable. Eso lo vemos en detalle en la siguiente etapa implementando `IEnumerable<T>` directamente.

---

## Etapa 6 — LINQ: consultas expresivas sobre conjuntos

En la última etapa, `Conjunto<T>` implementa `IEnumerable<T>` directamente. Con eso, *todo* LINQ está disponible sobre cualquier conjunto, gratis:

```csharp
// Ahora hacemes que el conjunto sea enumerable directamente, sin necesidad de la propiedad Elementos:
interface IConjunto<T> : IEnumerable<T> where T : IEquatable<T> {
    void Agregar(T valor);
    void Eliminar(T valor);
    bool Contiene(T valor);
}

class Conjunto<T> : IConjunto<T> where T : IEquatable<T> {
    readonly List<T> elementos = new();

    public void Agregar(T valor) { 
        if (!Contiene(valor)) { return; }
        elementos.Add(valor); 
    }
    public void Eliminar(T valor) => elementos.Remove(valor); // Remove usa Contains internamente
    public bool Contiene(T valor) => elementos.Contains(valor);

    // Clase interna para el enumerador. Recorre la lista y devuelve un elemento a la vez.
    class Enumerador : IEnumerator<T> {
        readonly List<T> elementos;
        private int index = -1;

        public Enumerador(List<T> elementos) {
            this.elementos = elementos;
        }

        public bool MoveNext() => ++index < elementos.Count;
        public T Current => elementos[index];
        public void Reset() => index = -1;

        object IEnumerator.Current => Current; // Por compatibilidad con la versión no genérica
    }

    public IEnumerator<T> GetEnumerator() => new Enumerador(elementos);
    IEnumerator IEnumerable.GetEnumerator() => GetEnumerator();
}
```

### Cómo funciona `GetEnumerator` con `yield return`

Hay una forma más sencilla de escribir el enumerador usando `yield return`:
El linguaje C# tiene una característica llamada **iteradores** que permite escribir enumeradores de forma más concisa usando `yield return`. En lugar de definir una clase `Enumerador` completa, podemos escribir el método `GetEnumerator` de esta manera:
El compilador se encarga de generar la clase enumeradora detrás de escena, y el código es mucho más directo: simplemente recorremos la lista y devolvemos cada elemento uno a uno.

```csharp
public IEnumerator<T> GetEnumerator() {
    foreach (var e in elementos) {
        yield return e; // devuelve un elemento a la vez
    }
}
```

`foreach` traduce a llamadas a `GetEnumerator()`. Nosotros podríamos devolver `elementos.GetEnumerator()` directamente, pero usamos `yield return` porque es la forma más declarativa: el compilador genera una máquina de estados que produce los elementos uno a uno.

La segunda línea, `IEnumerator IEnumerable.GetEnumerator()`, es una **implementación explícita** de la versión no genérica (anterior a los genéricos, heredada por compatibilidad). Reenvía a la versión tipada.

### LINQ en acción

Con `Conjunto<T>` siendo `IEnumerable<T>`, se habilitan de golpe todos los operadores de LINQ:

```csharp
// Estadísticas
Console.WriteLine($"Promedio: {teorica.Average(a => a.Nota):F2}");
Console.WriteLine($"Mínima:   {teorica.Min(a => a.Nota):F2}");
Console.WriteLine($"Máxima:   {teorica.Max(a => a.Nota):F2}");

// Filtrado y ordenamiento
foreach (var a in teorica.Where(a => a.Legajo > 20).OrderBy(a => a.Nombre)) {
    Console.WriteLine($" - {a}");
}

// Unión e intersección — ya provistas por LINQ
foreach (var a in teorica.Union(practica))     { /* en alguna */ }
foreach (var a in teorica.Intersect(practica)) { /* en ambas  */ }
```

Notar algo importante: **ya no necesitamos escribir `Union` e `Intersección` a mano**. LINQ los provee como métodos de extensión sobre cualquier `IEnumerable<T>`, y usan internamente el `Equals` / `GetHashCode` que definimos en la etapa 4. Todo el trabajo previo se compone.

### La restricción `IComparable<T>`

Agregamos `IComparable<T>` para habilitar orden. Mientras `IEquatable<T>` responde "¿son iguales?", `IComparable<T>` responde "¿cuál va primero?". Es lo que permite que `OrderBy` y `Min` / `Max` funcionen sin pasar un comparador explícito.

## Recapitulando el camino

| Etapa | Herramienta                             | Problema que resuelve                 |
|-------|-----------------------------------------|---------------------------------------|
| 1     | Array + `count` manual                  | Entender la mecánica de fondo         |
| 2     | Composición con `List<int>`             | Sacar el ruido de manejar capacidad   |
| 3     | Genéricos `Conjunto<T>` + interface     | Evitar duplicar la clase por tipo     |
| 4     | `IEquatable<T>`                         | Controlar qué significa "igualdad"    |
| 5     | `IEnumerable<T>` como propiedad         | Encapsular la estructura interna      |
| 6     | `IEnumerable<T>` implementado + `yield` | Integración total con LINQ            |

Cada refactor saca código y agrega expresividad. La clase final tiene menos líneas que la primera, hace muchísimo más, y el grueso del trabajo lo hacen las abstracciones del lenguaje — no código nuestro.

Esa es la enseñanza de fondo: en C# moderno, "saber programar" es en buena parte saber *qué interfaces implementar* para que el ecosistema (LINQ, `foreach`, colecciones, serialización) trabaje para nosotros.

## Resumen

```
¿Necesito una colección propia con unicidad?
    └─► `Conjunto<T>`

¿Quiero evitar duplicar código para `int`, `string` o `Alumno`?
    └─► genéricos (`Conjunto<T>`)

¿Necesito definir qué significa que dos elementos sean iguales?
    └─► `IEquatable<T>` y `Equals`

¿Quiero recorrer el conjunto con `foreach`?
    └─► `IEnumerable<T>` y `GetEnumerator()`

¿Prefiero generar elementos uno a uno sin guardar toda la lista?
    └─► `yield return`

¿Quiero combinar, filtrar u ordenar conjuntos con operaciones expresivas?
    └─► LINQ sobre `IEnumerable<T>`

¿Necesito comparar elementos para ordenar o sacar mínimos/máximos?
    └─► `IComparable<T>`

¿Quiero proteger la estructura interna del conjunto?
    └─► exponer `IEnumerable<T>`, no `List<T>`

¿Quiero que el ecosistema haga más trabajo por mí?
    └─► implementar las interfaces correctas
```
