# Strings en C#

`string` merece una sección propia porque es uno de los tipos más usados y, al mismo tiempo, uno de los que más engañan al principio.

## Qué es `string`

Representa texto.

```cs
string nombre  = "Ada";
string mensaje = "Hola mundo";
```

Aunque a simple vista parece "un conjunto de caracteres", en C# `string` es un tipo de referencia con comportamiento especial y muy optimizado.

---

## Formas de construir un string

### Concatenación

La forma más directa es usar `+`.

```cs
string saludo = "Hola, " + nombre;
```

Funciona, pero cuando hay muchos fragmentos se vuelve difícil de leer.

### Interpolación

La interpolación permite insertar expresiones dentro de una cadena usando `$`. Es la forma más legible para armar texto con valores.

```cs
string nombre = "Ada";
int edad = 20;

string mensaje = $"{nombre} tiene {edad} años";
```

Dentro de las llaves puede ir cualquier expresión válida de C#:

```cs
string resultado = $"El doble de {edad} es {edad * 2}";
```

### Verbatim strings

Los **verbatim strings** se escriben con `@` y permiten incluir saltos de línea literales y rutas sin escapar barras invertidas.

```cs
string ruta  = @"C:\Users\Ada\Documentos";

string texto = @"Línea 1
Línea 2
Línea 3";
```

En el segundo caso, los saltos de línea escritos en el código forman parte del contenido del string.

La forma alternativa sin `@` requiere usar secuencias de escape:

```cs
string texto = "Línea 1\nLínea 2\nLínea 3";
```

Ambas producen el mismo resultado. Con `@` el texto queda más parecido a cómo se verá; con `\n` el programador controla explícitamente dónde aparece cada salto.

También pueden combinarse `@` y `$`:

```cs
string carpeta = "Documentos";
string ruta = $@"C:\Users\Ada\{carpeta}";
```

# Raw string literals para escribir JSON

Una ventaja muy práctica de los raw string literals (""") es que permiten escribir JSON directamente en el código, con saltos de línea y comillas dobles sin tener que escaparlas.
```cs
string json = """
    {
        "nombre": "Ada",
        "edad": 20,
        "activo": true
    }
    """;
```
El texto del JSON queda idéntico a cómo se vería en un archivo .json. Eso lo hace cómodo para ejemplos, pruebas o textos de configuración pequeños.
Si además querés insertar valores de variables, usás $$""" en lugar de """, y las expresiones van entre {{ y }}:
```cs
string nombre = "Ada";
int edad = 20;
string json = $$"""
    {
        "nombre": "{{nombre}}",
        "edad": {{edad}}
    }
    """;
```
Con $$ las llaves simples { y } son texto literal, por eso el JSON no necesita ningún ajuste. Solo las expresiones interpoladas llevan doble llave.

Ahí siguen siendo necesarios los dobles `{{` y `}}` para escribir llaves literales dentro de una interpolación.

## Inmutabilidad

Una idea muy importante:

> Los `string` son **inmutables**.

Eso significa que, una vez creado un texto, su contenido no puede modificarse. Cuando parece que se cambia un `string`, en realidad se crea uno nuevo.

```cs
string nombre = "Ana";
nombre = nombre + " María";
```

No se edita el objeto original carácter por carácter: se genera un nuevo texto y la variable apunta a ese nuevo valor. El string `"Ana"` sigue existiendo en memoria hasta que el recolector de basura lo libere.

### `StringBuilder` para construcción intensiva

Cuando se construye un string dentro de un bucle con muchas concatenaciones, cada `+` genera un objeto nuevo. Para esos casos conviene usar `StringBuilder`, que trabaja sobre un buffer interno y solo genera el string final al terminar.

```cs
var sb = new StringBuilder();

for (int i = 0; i < 1000; i++)
{
    sb.Append($"Línea {i}\n");
}

string resultado = sb.ToString();
```

Para usos cotidianos, la concatenación y la interpolación son suficientes. `StringBuilder` es una optimización para escenarios de construcción intensiva.

## Conversión automática a `string`

En C#, muchos tipos pueden convertirse automáticamente a `string` cuando se usan en un contexto de texto.

```cs
int numero = 42;
string mensaje = "El número es: " + numero;
```

Esto ocurre porque todas las clases heredan de `object`, que tiene un método `ToString()`. Cuando se necesita texto, el sistema llama a ese método para obtener la representación textual del objeto.

Esto también aplica a las clases que definamos nosotros mismos, siempre que implementemos `ToString()`:

```cs
class Punto
{
    public int X { get; set; }
    public int Y { get; set; }

    public override string ToString() => $"({X}, {Y})";
}

var p = new Punto { X = 3, Y = 7 };
Console.WriteLine($"El punto es {p}"); // El punto es (3, 7)
```

## Comparación de strings

### Igualdad

Para comparar strings se usa `==`, que en C# compara el **contenido**, no la referencia:

```cs
string a = "hola";
string b = "hola";

Console.WriteLine(a == b); // true
```

Esto es distinto a lo que ocurre con otros tipos de referencia, donde `==` compara si dos variables apuntan al mismo objeto. Para `string`, `==` hace lo que uno espera intuitivamente.

### Orden: `CompareTo` y `string.Compare`

Para saber si un string va antes, después o igual que otro — útil al ordenar datos — se usa `CompareTo`. Devuelve un número negativo, cero o positivo:

```cs
string a = "banana";
string b = "manzana";

int resultado = a.CompareTo(b);
// resultado < 0 → "banana" va antes que "manzana"
// resultado = 0 → son iguales
// resultado > 0 → "banana" va después
```

La versión estática `string.Compare` hace lo mismo pero también acepta opciones de comparación:

```cs
int resultado = string.Compare("Ana", "ana", StringComparison.OrdinalIgnoreCase);
// 0 → iguales ignorando mayúsculas
```

### Ordenar una lista de strings

`List<T>.Sort()` usa `CompareTo` por defecto. Para tener control sobre cómo se comparan se pasa un `StringComparer`:

```cs
List<string> nombres = new List<string> { "pedro", "Ana", "luis", "Beatriz" };

// Orden por código ASCII: las mayúsculas van antes que las minúsculas
nombres.Sort();
// { "Ana", "Beatriz", "luis", "pedro" }

// Orden ignorando mayúsculas
nombres.Sort(StringComparer.OrdinalIgnoreCase);
// { "Ana", "Beatriz", "luis", "pedro" }  → mismo resultado aquí

// Orden según las reglas del idioma del sistema (incluye acentos, ñ, etc.)
nombres.Sort(StringComparer.CurrentCultureIgnoreCase);
```

La diferencia entre `Ordinal` y `CurrentCulture` importa cuando el texto incluye caracteres propios de un idioma. Por ejemplo, si los datos incluyen "ñ" o vocales acentuadas, `CurrentCulture` las ordena según las reglas del español; `Ordinal` las ordena por su valor numérico Unicode, lo que puede dar resultados inesperados.

### Comparación sin importar mayúsculas

Varios métodos aceptan un parámetro `StringComparison` para ignorar mayúsculas y minúsculas:

```cs
string frase = "El Cielo es Azul";

// Igualdad
bool igual = frase.Equals("el cielo es azul", StringComparison.OrdinalIgnoreCase); // true

// Búsqueda
bool contiene  = frase.Contains("cielo",  StringComparison.OrdinalIgnoreCase); // true
bool empieza   = frase.StartsWith("el",   StringComparison.OrdinalIgnoreCase); // true
bool termina   = frase.EndsWith("azul",   StringComparison.OrdinalIgnoreCase); // true
int  posicion  = frase.IndexOf("azul",    StringComparison.OrdinalIgnoreCase); // 13
```

La constante más usada es `StringComparison.OrdinalIgnoreCase` para comparaciones técnicas (nombres de archivo, claves, comandos). Para texto visible al usuario donde importan las reglas del idioma se prefiere `StringComparison.CurrentCultureIgnoreCase`.

El mismo `StringComparer` puede usarse al trabajar con diccionarios para que las claves sean insensibles a mayúsculas:

```cs
var config = new Dictionary<string, string>(StringComparer.OrdinalIgnoreCase);
config["Nombre"] = "Ada";

Console.WriteLine(config["nombre"]); // "Ada" — la clave se encuentra igual
```

## Null y string vacío

Un `string` puede ser `null` (no asignado) o vacío (asignado pero sin contenido). Son dos situaciones distintas.

```cs
string? nulo   = null;
string  vacio  = "";
string  blanco = "   ";
```

Para verificar si un string es `null` o vacío en un solo paso:

```cs
string.IsNullOrEmpty(nulo);   // true
string.IsNullOrEmpty(vacio);  // true
string.IsNullOrEmpty("Ada");  // false
```

Para detectar también strings que solo contienen espacios:

```cs
string.IsNullOrWhiteSpace(blanco); // true
string.IsNullOrWhiteSpace(vacio);  // true
```

En la práctica, `IsNullOrWhiteSpace` es más útil porque un string de solo espacios generalmente representa "sin contenido" desde el punto de vista del negocio.

## Métodos y propiedades comunes

### `Length`

Devuelve la cantidad de caracteres del string.

```cs
string nombre = "Ada";
int largo = nombre.Length; // 3
```

### `ToUpper` / `ToLower`

Convierten el texto a mayúsculas o minúsculas.

```cs
string texto = "Hola Mundo";

Console.WriteLine(texto.ToUpper()); // HOLA MUNDO
Console.WriteLine(texto.ToLower()); // hola mundo
```

### `Trim` / `TrimStart` / `TrimEnd`

Eliminan los espacios en blanco al inicio, al final, o en ambos extremos.

```cs
string con_espacios = "  hola  ";

Console.WriteLine(con_espacios.Trim());      // "hola"
Console.WriteLine(con_espacios.TrimStart()); // "hola  "
Console.WriteLine(con_espacios.TrimEnd());   // "  hola"
```

Útil para limpiar entradas del usuario antes de procesarlas.

### `Contains`

Verifica si un string contiene otro.

```cs
string frase = "El cielo es azul";

Console.WriteLine(frase.Contains("azul")); // true
Console.WriteLine(frase.Contains("rojo")); // false
```

### `StartsWith` / `EndsWith`

Verifican si el string empieza o termina con cierto texto.

```cs
string archivo = "informe_final.pdf";

Console.WriteLine(archivo.StartsWith("informe")); // true
Console.WriteLine(archivo.EndsWith(".pdf"));       // true
```

### `IndexOf`

Devuelve la posición de la primera aparición de un texto dentro del string. Si no existe, devuelve `-1`.

```cs
string frase = "hola mundo";

Console.WriteLine(frase.IndexOf("mundo")); // 5
Console.WriteLine(frase.IndexOf("chau"));  // -1
```

### `Replace`

Reemplaza todas las apariciones de un fragmento por otro.

```cs
string texto = "El gato y el gato";

string nuevo = texto.Replace("gato", "perro");
// "El perro y el perro"
```

### `Split`

Divide el string en un array de partes usando un separador.

```cs
string datos = "Ana,Luis,Pedro";

string[] nombres = datos.Split(',');
// { "Ana", "Luis", "Pedro" }
```

Puede usarse con cualquier carácter o string como separador.

### `string.Join`

Es la operación inversa a `Split`: une un array de strings usando un separador.

```cs
string[] nombres = { "Ana", "Luis", "Pedro" };

string resultado = string.Join(", ", nombres);
// "Ana, Luis, Pedro"
```

### `Substring`

Extrae una parte del string a partir de una posición, con longitud opcional.

```cs
string texto = "Programación III";

Console.WriteLine(texto.Substring(0, 12)); // "Programación"
Console.WriteLine(texto.Substring(13));    // "III"
```

En C# moderno también puede usarse la sintaxis de rangos, que es más compacta:

```cs
Console.WriteLine(texto[0..12]); // "Programación"
Console.WriteLine(texto[13..]);  // "III"
```

### `PadLeft` / `PadRight`

Agregan caracteres de relleno para que el string alcance cierta longitud. Útil para formatear salidas en columnas alineadas.

```cs
string numero = "42";

Console.WriteLine(numero.PadLeft(6));       // "    42"
Console.WriteLine(numero.PadLeft(6, '0'));  // "000042"
Console.WriteLine(numero.PadRight(6, '-')); // "42----"
```

## Unicode, `char` y emojis

Esta sección toca un tema que generalmente sorprende la primera vez que aparece en la práctica.

### Qué es Unicode

Unicode es el estándar que asigna un número único — llamado **code point** — a cada carácter del mundo: letras, dígitos, símbolos, caracteres de idiomas no latinos, emojis, y mucho más. Los code points van desde U+0000 hasta U+10FFFF, lo que da espacio para más de un millón de caracteres.

El problema es que no todos esos números caben en 16 bits. Ahí es donde aparece la diferencia entre un **code point** (el número que identifica al carácter) y una **unidad de codificación** (la representación en memoria).

### `char` en C# es una unidad de 16 bits

En C#, el tipo `char` almacena exactamente 16 bits. Eso alcanza para representar los primeros 65.536 code points de Unicode (U+0000 a U+FFFF), que incluyen prácticamente todos los caracteres del uso cotidiano en cualquier idioma occidental.

Pero los emojis, muchos caracteres históricos y símbolos matemáticos avanzados tienen code points mayores a U+FFFF. Para representarlos en UTF-16 (la codificación interna de `string` en C#) se usan **dos** `char` consecutivos llamados **par sustituto** (surrogate pair).

```cs
string emoji = "😀";            // U+1F600, está por encima de U+FFFF

Console.WriteLine(emoji.Length); // 2  ← no 1
```

`Length` cuenta **unidades de 16 bits**, no caracteres visibles. Para este emoji el resultado es `2`, aunque visualmente sea un solo carácter.

### El problema de recorrer con `foreach (char c in ...)`

El bucle estándar itera sobre `char`, lo que significa que un emoji queda partido en dos mitades ilegibles:

```cs
string texto = "A😀B";

foreach (char c in texto)
{
    Console.Write($"[{(int)c:X4}] ");
}
// [0041] [D83D] [DE00] [0042]
//   A     mitad1 mitad2   B
```

Las dos posiciones del medio son las dos mitades del surrogate pair. Si se intenta imprimir `c` directamente, C# puede mostrar un carácter de reemplazo o comportamiento indefinido.

### Recorrer correctamente con `Rune`

El tipo `Rune`, disponible desde .NET 5, representa un único code point Unicode completo. El método `EnumerateRunes()` devuelve un enumerador que une automáticamente los surrogate pairs:

```cs
string texto = "A😀B";

foreach (Rune r in texto.EnumerateRunes())
{
    Console.Write($"[{r}] ");
}
// [A] [😀] [B]
```

Cada `Rune` corresponde a un carácter real, sin importar cuántos `char` ocupa internamente. También expone el code point numérico:

```cs
Rune r = new Rune(0x1F600); // 😀
Console.WriteLine(r.Value); // 128512
```

### Longitud visible con `StringInfo`

`StringInfo` permite contar los **elementos de texto** (text elements), que corresponden a lo que el usuario ve como un carácter individual:

```cs
string texto = "Hola 😀";

Console.WriteLine(texto.Length);                           // 7
Console.WriteLine(new StringInfo(texto).LengthInTextElements); // 6
```

`Length` da 7 porque el emoji ocupa 2 `char`. `LengthInTextElements` da 6, que es la cantidad de caracteres visibles.

### Caracteres compuestos

Hay otro caso que también sorprende: un mismo carácter visible puede representarse de dos formas distintas en Unicode. Por ejemplo, la "é" puede ser:

- el code point U+00E9 (letra directamente),
- o la letra "e" (U+0065) seguida del acento combinado U+0301.

Ambas se ven igual pero son secuencias distintas de bytes:

```cs
string a = "\u00E9";       // é como un solo code point
string b = "e\u0301";     // é como e + acento combinante

Console.WriteLine(a == b); // false  (comparación ordinal)
Console.WriteLine(string.Compare(a, b, StringComparison.CurrentCulture)); // 0 (lingüísticamente iguales)
```

Esto se llama **normalización Unicode**. Para comparar texto de distintas fuentes de manera confiable se puede normalizar primero:

```cs
bool iguales = a.Normalize() == b.Normalize(); // true
```

### Qué tener en cuenta en la práctica

La mayoría de los programas cotidianos no necesitan preocuparse por esto: si el texto no contiene emojis ni mezcla fuentes con distinta normalización, `Length` y `foreach (char c in ...)` funcionan bien. Pero conviene conocer estas limitaciones porque:

- si se cuenta caracteres para validar una longitud visible (como en un chat o una bio), `Length` puede dar un número mayor al esperado,
- si se procesan archivos de texto de distintos orígenes, la normalización puede causar comparaciones falsamente negativas.

Para esos casos, `Rune` y `StringInfo` son las herramientas correctas.
